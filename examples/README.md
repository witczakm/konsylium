# Examples — real konsylium runs

These are **illustrative single runs** (model-mediated, date-stamped) — not canonical answers.
They show the *shape* of a verdict: one recommendation, preserved **dissent**, and an explicit
**"what we don't know"**. The same question can yield a different panel or different dissent —
that's divergence working as intended, not a defect.

| # | Decision | What it demonstrates |
|---|---|---|
| [01](01-build-vs-adopt.md) | Build our own council skill vs adopt an existing one | the council on a tooling decision; a real 3:1 dissent |
| [02](02-db-schema-pii.md) | Where to store bidder PII vs scoring (DB schema) | **the adaptive Marshal** minting Privacy / Compliance / Data-integrity personas — and catching an audit-snapshot requirement a generic panel would miss |
| [03](03-bash-vs-python.md) | Bash vs Python for a one-off script | a fast-path panel and an honest minority dissent |
| [04](04-oss-release-strategy.md) | How to release this project professionally | the council on **non-code strategy** (growth / skeptic panel) |

> Why these double as evidence: example **02** is the clearest proof of the headline claim — the
> Marshal chose data-integrity + privacy + compliance personas *because it read the question*, and
> two of them independently surfaced an immutable-snapshot requirement that an architect-only panel
> tends to miss. See the project README's "Honest limitations" for what these examples do **not** prove.
